## 🔗 关键协作节点

### 节点 1：Skills → 混合检索对接 (Week 3-4)

**交接方**: 金韩溢 → 卢周全

金韩溢开发的 Skills 框架提供了标准接口（`BaseSkill`），卢周全的混合检索模块应封装为 Skill 接入：

```python
# 卢周全的混合检索 Skill 应继承 BaseSkill
from vuln_analysis.skills.base import BaseSkill
from vuln_analysis.skills.registry import register_skill

@register_skill
class HybridSearchSkill(BaseSkill[SearchInput, SearchOutput, HybridSearchConfig]):
    name = "hybrid_search_skill"
    # ...
```

**协作要求**:
- 卢周全在开发检索模块时，参考 `skills/intel.py` 的实现模式
- 检索接口的输入/输出使用 Pydantic 模型定义
- 完成后通知金韩溢进行集成

## 🌿 Git 协作规范

### 分支策略

```
main                          # 稳定主分支 (只通过 PR 合并)
├── feature/multi-agent       # 金韩溢: 多 Agent 引擎
├── feature/knowledge-graph   # 卢周全: 知识图谱 + 检索
└── feature/test-cases        # 邓一凡: 测试用例 + 数据
```

### 提交规范

```bash
# 格式: <类型>: <描述>
git commit -m "feat: 实现 Supervisor Agent 状态机"
git commit -m "fix: 修复 SBOM 解析空指针异常"
git commit -m "test: 新增 Log4j 测试用例"
git commit -m "docs: 更新 API 文档"
```

| 前缀 | 含义 |
|:---|:---|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `test` | 测试相关 |
| `docs` | 文档更新 |
| `refactor` | 重构 |
| `chore` | 配置/杂项 |

### PR 流程

1. 在自己的 feature 分支上开发
2. push 后在 GitHub 上创建 Pull Request
3. 通知金韩溢 Review
4. Review 通过后合并到 main

## ⚠️ 重要约定

1. **不要直接 push 到 main** — 必须通过 PR，避免代码冲突。
2. **每个 commit 必须能通过测试** — push 前运行 `uv run python -m pytest tests/ -v`。
3. **代码文件头部加许可证** — 所有 `.py` 文件开头加 Apache 2.0 许可证注释。
4. **NVIDIA_API_KEY 不要提交到 Git** — 已在 `.gitignore` 中配置，切勿手动添加。
5. **有问题及时沟通** — 不要卡住超过半天不说。
