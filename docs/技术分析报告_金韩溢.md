### 2.1 `base.py` — Skill 的模板（抽象基类）

#### 这个文件有什么

一个叫 `BaseSkill` 的类，它是所有 Skill 的"父类模板"。

#### 它的结构

```python
class BaseSkill(abc.ABC, Generic[InputT, OutputT, ConfigT]):
    name: str              # Skill 名称，如 "intel_skill"
    description: str       # 描述，告诉 Agent 这个 Skill 能做什么
    input_schema: Type     # 输入数据的格式定义
    output_schema: Type    # 输出数据的格式定义
    config_schema: Type    # 配置项的格式定义

    def __init__(config, builder):  # 初始化时接收配置和 NAT 构建器
    async def run(input_data):      # 【必须实现】具体的执行逻辑
    def get_function_info():        # 把自己转换成 NAT 能识别的格式
```

#### 为什么这么设计

**问题**: NVIDIA 原始代码里，每个功能（查情报、解析 SBOM、搜代码）都是独立的 Python 函数，长得各不相同。`cve_fetch_intel.py` 的接口和 `cve_process_sbom.py` 完全不同，无法统一管理。

**解决**: 定义一个统一模板。不管是什么 Skill，都遵守同一套规则：
- 都有 `name` 和 `description`（方便 Agent 知道能用哪些工具）
- 都有 `input_schema` 和 `output_schema`（用 Pydantic 模型定义，类型安全）
- 都有一个 `run()` 方法（统一的调用方式）

**`Generic[InputT, OutputT, ConfigT]` 是什么**: 这是 Python 的"泛型"。意思是每个 Skill 可以有不同的输入输出类型，但框架统一按模板处理。比如 IntelSkill 的输入是 CVE 信息，ConfigSkill 的输入是 SBOM 数据，但框架不需要关心这些差异。

#### `get_function_info()` 方法详解

```python
def get_function_info(self) -> FunctionInfo:
    async def _wrapper(input_data):    # 包一层，加上日志和错误处理
        result = await self.run(input_data)
        return result

    return FunctionInfo.from_fn(       # 转成 NAT 的 FunctionInfo 格式
        _wrapper,
        input_schema=self.input_schema,
        description=self.description
    )
```

**作用**: NAT 框架不认识我们的 `BaseSkill`，它只认 `FunctionInfo` 对象。这个方法就是一个"翻译器"，把 Skill 翻译成 NAT 能用的格式。

### 2.2 `registry.py` — 自动注册机制

#### 这个文件有什么

一个叫 `register_skill` 的**装饰器函数**，只有 20 行代码。

#### 它怎么工作

```python
@register_skill                    # 加了这一行装饰器
class IntelSkill(BaseSkill[...]):  # 就会被自动注册到 NAT
    ...
```

内部做了什么：
```python
def register_skill(skill_cls):
    config_cls = skill_cls.config_schema    # 取出配置类

    @register_function(config_type=config_cls, ...)  # 注册到 NAT
    async def _skill_factory(config, builder):
        skill = skill_cls(config, builder)           # 创建 Skill 实例
        yield skill.get_function_info()              # 转成 NAT 格式

    return skill_cls  # 关键！返回原始类，不是工厂函数
```

#### 为什么 `return skill_cls` 很重要

这里有一个我们修复的 Bug。Python 装饰器的默认行为是：

```python
@decorator
class Foo:
    name = "foo"

# 等价于: Foo = decorator(Foo)
```

如果 `decorator` 返回的是内部的工厂函数 `_skill_factory`，那么 `Foo.name` 就访问不到了（因为 `Foo` 已经被替换成了 `_skill_factory`）。

**我们的修复**: 让装饰器返回原始的 `skill_cls`，这样类的属性（`name`、`description` 等）仍然可以正常访问。工厂函数虽然也注册了，但它是通过 `@register_function` 独立注册的，不影响类本身。

### 2.3 `intel.py` — 情报检索 Skill

#### 做什么

输入一组 CVE 编号，去 NVD（美国国家漏洞数据库）、GHSA（GitHub 安全公告）、RedHat、Ubuntu 等多个源查询每个 CVE 的详细情报。

#### 核心代码

```python
async def run(self, message):
    async with aiohttp.ClientSession(timeout=timeout) as session:
        intel_retriever = IntelRetriever(session=session, ...)
        tasks = [intel_retriever.retrieve(vuln_id=cve.vuln_id) 
                 for cve in message.input.scan.vulns]        # 每个 CVE 创建一个检索任务
        results = await asyncio.gather(*tasks)                # 并发执行所有检索
    message.info.intel = results                              # 把结果写入消息
```

#### 关键技术点

- **并发检索**: `asyncio.gather` 同时查询所有 CVE 的情报，而不是一个一个排队
- **重试机制**: 如果某个数据源超时或返回错误，会自动重试最多 5 次
- **超时控制**: 每个请求 30 秒超时，避免某个数据源卡住拖慢整个流程
- **底层调用**: 实际的 HTTP 请求由 NVIDIA 原始的 `IntelRetriever` 执行，我们做的是将其封装为 Skill 接口

### 2.4 `config.py` — SBOM 解析 Skill

#### 做什么

解析容器的 **SBOM（Software Bill of Materials，软件物料清单）**。SBOM 就是一个清单文件，记录了容器里安装了哪些软件包和版本号。

#### SBOM 文件长什么样

```
NAME        VERSION    TYPE
python      3.12.1     python
openssl     3.0.12     deb
log4j-core  2.14.1     java
```

#### 三种输入模式

```python
if sbom_info.type == "manual":       # 用户直接传入包列表
    packages = sbom_info.packages

elif sbom_info.type == "file":       # 从本地文件读取
    with open(sbom_info.file_path, "r", encoding="utf-8-sig") as f:
        sbom_lines = f.readlines()
    packages = self._parse_sbom_packages(sbom_lines)

elif sbom_info.type == "http":       # 从远程 URL 下载
    response = http_utils.request_with_retry(url=sbom_info.url, ...)
    sbom_lines = response.text.splitlines()
    packages = self._parse_sbom_packages(sbom_lines)
```

#### BOM 头 Bug 修复

```python
# NVIDIA 原始代码:
with open(file_path, "r", encoding="utf-8") as f:     # ❌ 会读到 BOM 头

# 我们的修复:
with open(file_path, "r", encoding="utf-8-sig") as f:  # ✅ 自动跳过 BOM 头
```

**什么是 BOM 头**: Windows 上的一些编辑器（如记事本）在保存 UTF-8 文件时，会在文件开头加入 3 个不可见字节 `\xef\xbb\xbf`（叫 BOM，Byte Order Mark）。用普通的 `utf-8` 编码读取时，这 3 个字节会被当成正文的一部分，导致第一行解析失败。`utf-8-sig` 编码会自动识别并跳过 BOM 头。

### 2.5 `remote_code.py` — 远程代码搜索 Skill（核心创新）

#### 做什么

在 GitHub 上的源码仓库中搜索特定关键词（通常是漏洞函数名），判断漏洞是否在代码层面可达。

#### 为什么是核心创新

NVIDIA 原始做法：
```
1. git clone 整个源码仓库 → 本地磁盘     (耗时、占空间)
2. 用 NVIDIA Embedder 将代码向量化       (慢、贵)
3. 存入 FAISS 向量数据库                  (复杂)
4. 用语义搜索查询漏洞函数                 (不精确)
```

我们的做法：
```
1. 调用 GitHub Code Search API 一次请求搞定  (快、免费、精确)
```

在 Windows 环境下，NVIDIA 的方案有一个**致命问题**: Git Clone 时路径长度超出 Windows 260 字符限制，直接崩溃。我们的方案完全绕过了这个问题。

#### 核心代码

```python
async def run(self, input_data):
    # 构造 GitHub 搜索查询
    search_query = f"{input_data.query}+repo:{input_data.repo}"
    url = f"https://api.github.com/search/code?q={search_query}"

    async with aiohttp.ClientSession() as session:
        async with session.get(url, headers=headers) as response:
            if response.status == 200:
                data = await response.json()
                # 解析搜索结果
                results = [CodeSearchResult(
                    file_path=item["path"],
                    content=item["name"],
                    repo=item["repository"]["full_name"]
                ) for item in data["items"]]
                return RemoteCodeSearchOutput(results=results, total_count=data["total_count"])
```

#### 错误处理

```python
status == 401 → "认证失败，检查 GITHUB_TOKEN"
status == 403 → "API 速率限制，稍后再试"
status == 422 → "仓库不存在或为私有"
其他           → "未预期的错误"
```

#### 使用示例

```python
input = RemoteCodeSearchInput(
    query="email.utils.parseaddr",   # 搜索 CVE-2023-36632 的漏洞函数
    repo="NVIDIA/morpheus",          # 在这个仓库里搜
    max_results=10
)
output = await remote_code_skill.run(input)
# output.total_count = 3  → 仓库中有 3 处调用了这个函数
# output.results[0].file_path = "src/utils/email_parser.py"
```

### 3.1 `state.py` — 共享状态（Agent 之间的黑板）

#### 做什么

定义了所有 Agent 之间传递信息用的数据格式。可以理解为一块"黑板"，每个 Agent 在上面写自己的发现，其他 Agent 可以读。

#### 6 个数据模型

```python
class IntelResult:          # Intel Agent 的输出
    cve_id: str             # CVE 编号
    severity: str           # 严重程度 (critical/high/medium/low)
    description: str        # 漏洞描述

class CodeSearchResult:     # Code Agent 的输出
    cve_id: str
    code_found: bool        # 源码中是否找到了漏洞函数调用
    matched_files: list     # 匹配到的文件列表
    evidence: str           # 证据描述

class ConfigResult:         # Config Agent 的输出
    cve_id: str
    package_found: bool     # SBOM 中是否包含漏洞包
    package_name: str       # 漏洞包名
    is_vulnerable_version: bool  # 是否为受影响版本

class VEXJudgment:          # VEX Agent 的最终判定
    cve_id: str
    status: str             # 判定: "affected" / "not_affected" / "unknown"
    justification: str      # 理由
    confidence: float       # 置信度 0-1

class MultiAgentState:      # 全局状态（黑板）
    cve_list: list[str]                           # 待分析的 CVE 列表
    intel_results: dict[str, IntelResult]          # 情报结果
    code_results: dict[str, CodeSearchResult]      # 代码搜索结果
    config_results: dict[str, ConfigResult]        # 配置检查结果
    vex_judgments: dict[str, VEXJudgment]          # 最终判定
    current_phase: str                             # 当前阶段
    errors: list[str]                              # 错误记录
```

#### 为什么要单独定义状态

如果不定义，就会出现这种混乱：
- Intel Agent 返回 `{"severity": "high"}`
- Code Agent 期望收到 `{"level": "高"}`
- 格式不统一，对接就出 Bug

有了统一的 Pydantic 模型，每个字段的名称、类型、默认值都是明确的，编辑器和测试都能自动校验。

### 3.2 `supervisor.py` — Supervisor Agent（调度中心）

#### 做什么

用 LangGraph 构建一个四步状态机，调度子 Agent 的执行顺序。

#### 四个节点

```python
# 节点 1: init（初始化）
async def init_node(state):
    state.cve_list = [v.vuln_id for v in state.engine_input.input.scan.vulns]
    # 从输入中提取要分析的 CVE 列表
    return state

# 节点 2: gather（信息收集 — 三路并行）
async def gather_node(state):
    intel_state, code_state, config_state = await asyncio.gather(
        intel_agent_node(state复制),   # Intel Agent 同时执行
        code_agent_node(state复制),    # Code Agent 同时执行
        config_agent_node(state复制),  # Config Agent 同时执行
    )
    # 合并三个 Agent 的结果到主状态
    state.intel_results = intel_state.intel_results
    state.code_results = code_state.code_results
    state.config_results = config_state.config_results
    return state

# 节点 3: judge（综合判定）
async def judge_node(state):
    state = await vex_agent_node(state)  # VEX Agent 综合三方证据
    return state

# 节点 4: summary（输出汇总）
async def summary_node(state):
    # 统计并输出 affected / not_affected / unknown 的数量
    return state
```

#### 状态机构建

```python
def build_supervisor_graph():
    graph = StateGraph(MultiAgentState)

    graph.add_node("init", init_node)
    graph.add_node("gather", gather_node)
    graph.add_node("judge", judge_node)
    graph.add_node("summary", summary_node)

    # 边的连接: START → init → gather → judge → summary → END
    graph.add_edge(START, "init")
    graph.add_edge("init", "gather")
    graph.add_edge("gather", "judge")
    graph.add_edge("judge", "summary")
    graph.add_edge("summary", END)

    return graph
```

#### 为什么用 LangGraph

LangGraph 是 LangChain 团队出的状态机框架，专门用于构建 AI Agent 工作流。它的好处：
- **可视化**: 可以自动生成流程图
- **可恢复**: 如果中间某步失败，可以从断点恢复
- **可扩展**: 后续可以加条件边（比如"如果情报不足，跳过代码搜索"）

### 3.4 `code_agent.py` — 代码审计员

#### 做什么

判断漏洞函数在源码中是否被调用。当前版本使用模式匹配表（后续会集成 RemoteCodeSkill 调用 GitHub API）。

#### 已知漏洞模式表

```python
CVE_SEARCH_PATTERNS = {
    "CVE-2023-36632": ["email.utils.parseaddr", "parseaddr"],
    "CVE-2021-44228": ["org.apache.logging.log4j", "JndiLookup"],
    "CVE-2022-22965": ["spring-webmvc", "ClassPathResource"],
    "CVE-2014-0160":  ["SSL_read", "dtls1_process_heartbeat"],
}
```

#### 关键词生成逻辑

```python
def _get_search_queries(cve_id, intel_description):
    if cve_id in CVE_SEARCH_PATTERNS:      # 如果有已知模式
        return CVE_SEARCH_PATTERNS[cve_id]  # 直接用
    
    # 否则从情报描述中提取可能的函数名
    for word in intel_description.split():
        if '.' in word and len(word) > 5:   # 带小数点的长词可能是函数名
            queries.append(word)            # 如 "org.apache.log4j"
    
    return queries if queries else [cve_id] # 最后兜底用 CVE ID 本身
```

### 3.5 `config_agent.py` — 配置审计员

#### 做什么

检查容器的 SBOM 包列表中是否包含漏洞相关的软件包。

#### 核心逻辑

```python
# 1. 获取 SBOM 中所有包名
package_names = {pkg.name.lower() for pkg in packages}  # 如 {"python", "openssl", "log4j-core"}

# 2. 对每个 CVE，检查情报中提到的受影响包是否在 SBOM 里
for cve_id in state.cve_list:
    intel = state.intel_results.get(cve_id)
    
    # 方法一: 从情报的 affected_packages 字段精确匹配
    for pkg in intel.intel_data.get('affected_packages', []):
        if pkg['name'].lower() in package_names:
            package_found = True
    
    # 方法二: 如果没有精确数据，用模糊匹配（漏洞描述中提到的包名）
    if not package_found:
        for pkg in packages:
            if pkg.name.lower() in intel.description.lower():
                package_found = True
```

### 3.6 `vex_agent.py` — 最终裁判

#### 做什么

综合 Intel / Code / Config 三个 Agent 的结果，对每个 CVE 做出最终判定。

#### 判定逻辑（决策树）

```
                    有情报吗？
                   /         \
                 否            是
                 ↓             ↓
            unknown        SBOM 中有漏洞包吗？
            (20%)          /              \
                         否                是
                         ↓                 ↓
                   not_affected       代码中有漏洞函数吗？
                   (85%)              /              \
                                    否                是
                                    ↓                 ↓
                              not_affected       affected
                              (70%)              (90%)
```

#### 对应代码

```python
def _judge_vex_status(cve_id, has_intel, severity, code_found, package_found, is_vulnerable):
    if not has_intel:
        return "unknown", "情报不足", 0.2

    if not package_found:
        return "not_affected", "SBOM 中未发现受影响的包", 0.85

    if package_found and not code_found:
        return "not_affected", "包存在但代码不可达", 0.7

    if package_found and code_found and is_vulnerable:
        return "affected", "包存在且代码可达", 0.9

    return "unknown", "需要人工审查", 0.3
```

#### VEX 是什么

VEX = Vulnerability Exploitability eXchange（漏洞可利用性交换格式）。是一个行业标准，用于说明"这个漏洞在我的产品中到底有没有影响"。

传统漏洞扫描器（如 Trivy）只会告诉你"有 100 个 CVE"，但不会告诉你哪些真正影响了你的项目。VEX 就是用来补充这个信息的。**这正是 ContainerGuard AI 的核心价值。**
