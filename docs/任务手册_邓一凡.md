## 📦 任务一：准备被测容器镜像（阶段二前半，2.23 - 3.1）

### 1.1 目标
准备一组**包含已知漏洞的容器镜像**，作为系统的标准测试集。

### 1.2 需要准备的镜像列表

| # | 镜像 | 目标 CVE | 漏洞类型 | 说明 |
|:---|:---|:---|:---|:---|
| 1 | `morpheus:23.11-runtime` | CVE-2023-36632 | Python cpython DoS | 项目自带的测试镜像 |
| 2 | Log4j 漏洞镜像 | CVE-2021-44228 | Java RCE | Log4Shell，史上最严重漏洞之一 |
| 3 | Spring4Shell 镜像 | CVE-2022-22965 | Java RCE | Spring Framework 远程代码执行 |
| 4 | OpenSSL 心脏出血镜像 | CVE-2014-0160 | C 内存泄漏 | Heartbleed |
| 5 | 综合漏洞镜像 | 多个 CVE | 混合 | 50+ CVE 的大型镜像（压力测试用） |

### 1.3 步骤

**第一步：了解输入格式**

查看项目已有的输入文件，了解格式：
```bash
# 查看现有的输入消息格式
cat src/vuln_analysis/data/input_messages/morpheus_23.11-runtime.json
```

输入 JSON 的关键结构：
```json
{
  "image": {
    "name": "镜像名称",
    "tag": "版本标签",
    "sbom_info": {
      "type": "file",
      "file_path": "SBOM文件路径"
    }
  },
  "scan": {
    "vulns": [
      {"vuln_id": "CVE-2021-44228"},
      {"vuln_id": "CVE-2022-22965"}
    ]
  }
}
```

**第二步：获取 SBOM 文件**

方法 A：使用 Syft 扫描 Docker 镜像生成 SBOM
```bash
# 安装 Syft (SBOM 生成工具)
# Windows: 从 https://github.com/anchore/syft/releases 下载
# 或用 scoop: scoop install syft

# 生成 SBOM
syft docker:log4j-vulnerable-app:latest -o syft-table > sboms/log4j.sbom
```

方法 B：手动构建 SBOM（如果没有 Docker 环境）
```
NAME VERSION TYPE
log4j-core 2.14.1 java-archive
spring-webmvc 5.3.14 java-archive
tomcat-embed-core 9.0.56 java-archive
```

**第三步：获取漏洞扫描结果**

方法 A：使用 Grype 扫描
```bash
# 安装 Grype (漏洞扫描工具)
# Windows: 从 https://github.com/anchore/grype/releases 下载

# 扫描
grype docker:log4j-vulnerable-app:latest -o json > scans/log4j_scan.json
```

方法 B：手动编写扫描结果
从 NVD (https://nvd.nist.gov/) 查询对应 CVE，手动填入 JSON。

**第四步：创建输入文件**

在 `src/vuln_analysis/data/input_messages/` 目录下，为每个镜像创建输入 JSON：

```
src/vuln_analysis/data/
├── input_messages/
│   ├── morpheus_23.11-runtime.json    # 已有
│   ├── log4j_vulnerable.json          # 新增
│   ├── spring4shell_vulnerable.json   # 新增
│   └── heartbleed_vulnerable.json     # 新增
├── sboms/
│   ├── morpheus_v23.11.01-runtime.sbom  # 已有
│   ├── log4j_vulnerable.sbom            # 新增
│   ├── spring4shell_vulnerable.sbom     # 新增
│   └── heartbleed_vulnerable.sbom       # 新增
```

### 1.4 交付物
- 至少 **3 个新的输入 JSON** + 对应 SBOM 文件
- 每个镜像至少包含 1 个已知 CVE
- 在飞书更新你的进度

## 📋 任务二：设计测试用例表（阶段二后半，3.1 - 3.8）

### 2.1 目标
设计一份完整的测试用例表，覆盖系统的各个层面。

### 2.2 测试用例分类

请创建文件 `tests/test_cases.md`，按以下 5 个维度设计测试用例：

#### A. 功能测试

| 用例 ID | 测试项 | 输入 | 期望输出 | 实际结果 | 状态 |
|:---|:---|:---|:---|:---|:---|
| F-001 | SBOM 解析正确性 | morpheus SBOM 文件 | 解析出 N 个包 | | |
| F-002 | CVE 情报检索 | CVE-2023-36632 | 返回 NVD/GHSA 数据 | | |
| F-003 | 远程代码搜索 | "parseaddr" in cpython | 找到匹配文件 | | |
| F-004 | 完整流水线 | morpheus 输入 JSON | 生成分析报告 | | |
| F-005 | 多 CVE 批量分析 | 含 5+ CVE 的输入 | 每个 CVE 都有判定 | | |

#### B. 可达性测试（Reachability）

| 用例 ID | 测试项 | CVE | 期望判定 | 说明 |
|:---|:---|:---|:---|:---|
| R-001 | 漏洞代码存在且可达 | CVE-2023-36632 | affected | cpython 中使用了 parseaddr |
| R-002 | 漏洞代码存在但不可达 | (待定) | not_affected | 包存在但未调用漏洞函数 |
| R-003 | 漏洞包不在 SBOM 中 | (待定) | not_affected | 根本没装该包 |

#### C. 知识图谱测试

| 用例 ID | 测试项 | 输入 | 期望输出 |
|:---|:---|:---|:---|
| K-001 | CVE→CWE 映射 | CVE-2023-36632 | CWE-674 |
| K-002 | CWE→CAPEC 映射 | CWE-674 | 对应攻击模式 |
| K-003 | 同类漏洞关联 | CWE-674 | 返回相关 CVE 列表 |

#### D. 集成测试

| 用例 ID | 测试项 | 说明 |
|:---|:---|:---|
| I-001 | 多 Agent 协同 | Supervisor 正确分发任务给子 Agent |
| I-002 | Skill 调用链 | Agent → Skill → 外部 API → 返回结果 |
| I-003 | 错误传播 | API 超时时 Agent 能优雅处理 |

#### E. 性能测试

| 用例 ID | 测试项 | 基准值 | 目标值 |
|:---|:---|:---|:---|
| P-001 | 单 CVE 分析时间 | (待测) | < 60s |
| P-002 | 10 CVE 批量时间 | (待测) | < 5min |
| P-003 | 50+ CVE 压力测试 | (待测) | 不崩溃，全部完成 |
| P-004 | Token 消耗 | (待测) | 记录统计 |

### 2.3 交付物
- `tests/test_cases.md` 完整测试用例表
- 至少填写 15 条测试用例

## 🧪 任务三：执行端到端测试（阶段三，3.9 - 3.22）

### 3.1 目标
在系统集成完成后，执行完整的端到端测试。

### 3.2 步骤

**第一步：运行基准测试**

```bash
# 使用项目自带的 morpheus 镜像运行
nat run --config_file=src/vuln_analysis/configs/config-local.yml \
        --input_file=src/vuln_analysis/data/input_messages/morpheus_23.11-runtime.json
```

**第二步：运行你准备的测试镜像**

```bash
# Log4j 镜像
nat run --config_file=src/vuln_analysis/configs/config-local.yml \
        --input_file=src/vuln_analysis/data/input_messages/log4j_vulnerable.json

# Spring4Shell 镜像
nat run --config_file=src/vuln_analysis/configs/config-local.yml \
        --input_file=src/vuln_analysis/data/input_messages/spring4shell_vulnerable.json
```

**第三步：记录结果**

每次运行后，记录：
1. 是否成功完成（✅ / ❌）
2. 分析报告是否生成
3. CVE 判定结果是否合理
4. 运行耗时
5. 遇到的错误信息（如有）

**第四步：Bug 报告**

在 GitHub Issues 中记录发现的 Bug，格式：
```
标题: [BUG] 简要描述
环境: Windows 11 / Python 3.12
复现步骤:
1. ...
2. ...
期望行为: ...
实际行为: ...
错误日志: (粘贴关键日志)
```

### 3.3 交付物
- 测试执行记录表（在 `tests/test_cases.md` 中填写实际结果列）
- GitHub Issues 中的 Bug 报告
- Bug 修复协助（与金韩溢协作）

## 📊 任务四：压力测试与数据收集（阶段四，3.23 - 4.5）

- 用 **50+ CVE 混合容器**做压力测试

### 4.2 压力测试步骤

1. 使用综合漏洞镜像（任务一中准备的第 5 个镜像）

编写 `docs/测试报告.md`，包含：
- 测试环境说明

- Bug 统计（发现/修复/遗留）
- 结论与改进建议

### 4.4 交付物
- `docs/测试报告.md`
- 实验数据（CSV 或 JSON 格式）

## 🛠️ 环境搭建

```bash
# 1. 克隆仓库
git clone https://github.com/King-Hanyi/containerguard-ai.git
cd containerguard-ai

# 2. 安装 uv
pip install uv

# 3. 安装依赖
uv sync

# 4. 配置环境变量
cp .env.template .env
# 编辑 .env，填入 NVIDIA_API_KEY (找金韩溢要)

# 5. 验证环境
uv run python -c "import vuln_analysis; print('环境OK')"

# 6. 运行现有测试，确认环境正常
uv run python -m pytest tests/test_skills.py -v
```

## ⚠️ 注意事项

1. **没有 Docker 也能做**: 你可以手动编写 SBOM 和输入 JSON，不一定需要真实 Docker 镜像。
2. **优先级**: 先完成 **准备测试输入文件**（任务一），再设计测试用例表（任务二）。
3. **代码提交**: 在 `feature/test-cases` 分支上开发，完成后 PR 到 main。
4. **沟通**: 测试相关的问题在飞书群里沟通；发现的 Bug 记录到 GitHub Issues。
